\begin{abstract}

This project attempts to build a user-level task scheduler that can distribute
tasks across parallel OCaml domains and use the observations across them to measure
the efficiency of work-stealing across domains. Work-stealing is a paradigm
where idle domains i.e. domains that have completed tasks allotted to them steal
tasks from other domains, thus rebalancing loads across domains and speeding up
overall program execution. At the same time, this introduces contention with
possibly multiple domains trying to steal from a domain that may be about to
pick up the same task for execution locally.

[paper] originally introduced this idea of using array-based deques (double-ended
queues) for managing tasks across multiprocessors. With fixed-size arrays,
inefficient memory usage and array overflow were the fundamental problems. Our
implementation is based on [Chase-Lev] which addressed both problems using
dynamic cyclic arrays:  arrays can grow or shrink as needed and using a cyclic
array ensures the overflow problem is only constrained by the number of bits
used for arrays.

\section{Goals}
\label{sec:goals}
% =============================================================================


The goals of the project are as follows:

\begin{enumerate}
    \item To implement the Chase-Lev work-stealing deque in OCaml: [paper]
     proposes a new data-structure: dynamic circular arrays that grow when
     tasks exceed the current array size and shrink when the number of elements
     in the array is less than the array size by a specific factor. We want to
     port the basic methods presented in the paper -- pop, push and steal --
     from Java to OCaml. For ensuring that deque stealing is
     guaranteed to be done only by one domain, OCaml's Atomic module, especially
     it's compare$\_$and$\_$set instruction will be crucial to our work.

   \item To implement a user-level parallel task scheduler: This
    scheduler will take a recursively computable problem and divide it into
    a set of initial tasks to be distributed across the scheduler work loops.
    Each work-loop in the parallel scheduler:
    \begin{enumerate}
        \item will be mapped to a deque (double-ended queue) to which these tasks
            will be pushed into.
        \item will spawn new subtasks that will be pushed into the local deque. When
            the scheduler is ready, tasks will be popped from the same end
            This end is the bottom end of the deque.
        \item can have tasks stolen from it by another domain but only
            from the end opposite to where tasks are being pushed and popped.
            This end is called top. The challenge will be to ensure this occurs
            correctly among possibly several domains trying to steal tasks from
            this work deque and to measure stealing for different classes of
            problems (memory-bound vs compute bound).
     \end{enumerate}
     The scheduler will support a fork command to break down the problem tasks
     into smaller-sized ones; to gather the computed results it will support
     a join command.

     Scheduler domains/deques can steal from each other using different
     strategies. The easiest strategy one can implement is randomly picking up
     a domain other than itself. Another one, suggessted in the problem statement
     given to us, is a round-robin system where each domain attempts to steal
     from the next one in a specified order. We will implement both and measure
     outputs for both strategies.

   \item To develop accurate models for the deque used in the implementation
       that can be used in checking correctness in the linearizability of the
       implementation using QCheck-Lin and sequential specification using
       QCheck-STM.

   \item To check that there are no data races ina the codebase, we will check
       all executions on an OCaml compiler instance augumented by TSAN
       (ThreadSanitizer).

    The following are the most interesting and possibly challenging parts of our
    goals:
    \begin{itemize}
        \item correctly implementing the growing, shrinking of the underlying
            circular arrays while ensuring correct deque behaviour - LIFO
            pushing and popping of tasks in a deque contends in a linearizable
            way with FIFO (stealing) of tasks from the deque by another domain.
        \item building a parallel scheduler that operates correctly with
            per-domain deques that spawn, join and steal tasks concurrently and
            correctly. We have encountered both unicore (single-domain) and
            parallel round-robin schedulers (effect-based) with a single shared
            task queue as the work loop in CS6868 lectures 9 and 10 but without
            any work-stealing. Ensuring the correctness of work stealing by a
            single domain among several competing domains
            in the parallel scheduler with dynamic arrays will be the key.
    \end{itemize}

   We want to tackle four problems either compute-bound (where execution speed
   is primarily determined by the speed with which the tasks the problem is
   divided into is executed by CPU) or memory-bound (where execution speed is
   primarily determined by the rate with which data is read from/written to the
   memory), which are:
   \begin{enumerate}
       \item parallel mergesort
       \item fibonacci
       \item <>
       \item <>
   \end{enumerate}

   and measure:
   \begin{enumerate}
       \item program speedup
       \item steal frequency
       \item task throughput
       \item the impact of steal policy on steal frequency (round-robin,
           random victim)
       \item the threshold or task size at which sequential execution
             is faster than parallel execution
   \end{enumerate}
   for both the parallel scheduler and an implementation of a mutex-protected
   shared-queue scheduler.


\end{enumerate}
